Visión General del Despliegue

Todo lo que necesitas saber antes de empezar a subir tu proyecto al servidor.

El objetivo es tener todos los contenedores corriendo de forma estable en un VPS (servidor Linux sin GPU), accesibles desde internet a través de Nginx. Esta guía cubre cada paso desde cero.

📐 Arquitectura que debes desplegar

🏆 Componentes evaluados en la rúbrica

ComponentePtsPara obtener puntaje completo…
🐳 Infraestructura Docker 0.7 docker compose up sin errores. Todos los servicios healthy. MinIO con bucket. Nginx con rate-limit. Volúmenes persistentes.
🔗 Herencia Corte 1 0.6 Doble API-Key, 3 roles, FHIR Patient+Observation, paginación, rate-limit 429, cifrado pgcrypto — funcionando en endpoints nuevos y antiguos.
🌱 Pacientes sintéticos 0.4 Script crea ≥ 30 pacientes con Observations LOINC + ≥ 15 con imagen en MinIO + recurso Media FHIR. Datos coherentes con el dataset real.

ℹ️
¿Por qué CPU-only? El VPS típico no tiene GPU. Por eso los modelos de ML y DL deben exportarse a formato ONNX con cuantización INT8 — esto reduce el tiempo de inferencia dramáticamente y elimina la dependencia de CUDA/PyTorch en producción.

Requisitos del Servidor VPS

Especificaciones mínimas y software que debe tener el servidor antes de empezar.

⚙️ Especificaciones mínimas del VPS

🔧 Hardware mínimo

  • CPU: 2 vCPU (x86_64)
  • RAM: 4 GB (recomendado 6-8 GB)
  • Disco: 20 GB SSD mínimo
  • Red: IP pública + puerto 80/443 abierto
  • GPU: ❌ No requerida (ONNX + CPU)

🐧 Sistema Operativo

  • OS: Ubuntu 22.04 LTS (recomendado)
  • Alternativa: Ubuntu 20.04 o Debian 11
  • Acceso: SSH con usuario sudo
  • Puertos: 22 (SSH), 80 (HTTP), 443 (HTTPS)

Instalar Docker y Docker Compose

▶
Instalar Docker Engine
# Instalar dependencias
sudo apt install -y ca-certificates curl gnupg lsb-release

# Agregar GPG key oficial de Docker
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
  sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# Agregar repositorio Docker
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Instalar Docker Engine
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin

# Verificar instalación
docker --version          # Docker version 25.x
docker compose version    # Docker Compose version v2.x
Configurar Docker sin sudo (recomendado)
# Agregar tu usuario al grupo docker
sudo usermod -aG docker $USER

# Aplicar cambio (cierra y vuelve a conectar por SSH)
newgrp docker

# Verificar: debería funcionar sin sudo
docker ps
⚠️
Si el VPS es de Render, Railway, o similar (no bare metal), Docker ya puede estar disponible. Verifica con docker --version antes de instalar.

🐳 docker-compose.yml — Estructura completa

Este es el archivo central que levanta todos los servicios. Cópialo y ajusta los nombres de imagen a los de tu proyecto:

version: '3.9'

services:

  # ── 1. BASE DE DATOS PRINCIPAL ──────────────────────────────────
  postgres:
    image: postgres:15-alpine
    container_name: c2-postgres
    restart: unless-stopped
    environment:
      POSTGRES_DB: ${POSTGRES_DB}
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    networks:
      - backend-net
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER}"]
      interval: 10s
      timeout: 5s
      retries: 5

  # ── 2. HAPI FHIR R4 ────────────────────────────────────────────
  hapi-fhir:
    image: hapiproject/hapi:v7.2.0
    container_name: c2-hapi-fhir
    restart: unless-stopped
    environment:
      spring.datasource.url: jdbc:postgresql://postgres:5432/${POSTGRES_DB}
      spring.datasource.username: ${POSTGRES_USER}
      spring.datasource.password: ${POSTGRES_PASSWORD}
      hapi.fhir.fhir_version: R4
      hapi.fhir.allow_external_references: "true"
    depends_on:
      postgres:
        condition: service_healthy
    networks:
      - backend-net
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/fhir/metadata"]
      interval: 30s
      timeout: 10s
      retries: 5
      start_period: 60s

  # ── 3. MINIO ───────────────────────────────────────────────────
  minio:
    image: minio/minio:latest
    container_name: c2-minio
    restart: unless-stopped
    command: server /data --console-address ":9001"
    environment:
      MINIO_ROOT_USER: ${MINIO_ROOT_USER}
      MINIO_ROOT_PASSWORD: ${MINIO_ROOT_PASSWORD}
    volumes:
      - minio_data:/data
    networks:
      - backend-net
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:9000/minio/health/live"]
      interval: 30s
      timeout: 5s
      retries: 3

  # ── 4. MLFLOW ─────────────────────────────────────────────────
  mlflow:
    image: ghcr.io/mlflow/mlflow:v2.13.0
    container_name: c2-mlflow
    restart: unless-stopped
    command: mlflow server --host 0.0.0.0 --port 5000 --backend-store-uri sqlite:///mlflow.db
    networks:
      - backend-net

  # ── 5. MAILHOG ────────────────────────────────────────────────
  mailhog:
    image: mailhog/mailhog:latest
    container_name: c2-mailhog
    restart: unless-stopped
    networks:
      - backend-net

  # ── 6. FASTAPI BACKEND ────────────────────────────────────────
  backend:
    build: ./backend
    container_name: c2-backend
    restart: unless-stopped
    env_file: .env
    depends_on:
      postgres:
        condition: service_healthy
      hapi-fhir:
        condition: service_healthy
      minio:
        condition: service_healthy
    networks:
      - backend-net
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 20s
      timeout: 5s
      retries: 3

  # ── 7. ML SERVICE (TABULAR) ──────────────────────────────────
  ml-service:
    build: ./ml-service
    container_name: c2-ml-service
    restart: unless-stopped
    env_file: .env
    networks:
      - backend-net
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8001/health"]
      interval: 20s
      timeout: 5s
      retries: 3

  # ── 8. DL SERVICE (IMAGEN) ───────────────────────────────────
  dl-service:
    build: ./dl-service
    container_name: c2-dl-service
    restart: unless-stopped
    env_file: .env
    networks:
      - backend-net
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8002/health"]
      interval: 20s
      timeout: 5s
      retries: 3

  # ── 9. ORQUESTADOR ────────────────────────────────────────────
  orchestrator:
    build: ./orchestrator
    container_name: c2-orchestrator
    restart: unless-stopped
    env_file: .env
    depends_on:
      - backend
      - ml-service
      - dl-service
    networks:
      - backend-net

  # ── 10. FRONTEND SPA ─────────────────────────────────────────
  frontend:
    build: ./frontend
    container_name: c2-frontend
    restart: unless-stopped
    networks:
      - backend-net

  # ── 11. NGINX (ENTRADA ÚNICA) ─────────────────────────────────
  nginx:
    image: nginx:alpine
    container_name: c2-nginx
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - backend
      - frontend
    networks:
      - backend-net

networks:
  backend-net:
    driver: bridge

volumes:
  postgres_data:
  minio_data:

Levantar los Servicios

El momento clave: docker compose up. Hay un orden importante para evitar errores de arranque.

🔢 Orden de arranque (automático con depends_on)

1. postgres

Base de datos. Debe estar healthy antes de que arranquen el backend y HAPI FHIR.

2. minio + mailhog + mlflow

Servicios de soporte. Arrancan en paralelo con postgres.

3. hapi-fhir

Tarda 1-2 minutos en inicializar (es Java). Espera hasta que responda en /fhir/metadata.

4. backend + ml-service + dl-service

Arrancan cuando postgres y minio están healthy.

5. orchestrator + frontend

Arrancan cuando el backend está listo.

6. nginx

Último en arrancar. Es la puerta de entrada al sistema.

▶️ Comandos de arranque

Primer arranque

Levantar todos los servicios

▶ 
# Arrancar todo en background (modo daemon)
docker compose up -d

# Ver el estado de todos los servicios
docker compose ps

# Salida esperada (puede tardar ~3 minutos hasta que hapi-fhir sea healthy):
# NAME               STATUS                      PORTS
# c2-postgres        Up (healthy)
# c2-minio           Up (healthy)
# c2-hapi-fhir       Up (healthy)
# c2-backend         Up (healthy)
# c2-ml-service      Up (healthy)
# c2-dl-service      Up (healthy)
# c2-orchestrator    Up
# c2-frontend        Up
# c2-nginx           Up                          0.0.0.0:80->80/tcp
Ver logs

Monitorear el arranque

▶ 
# Ver logs de todos en tiempo real
docker compose logs -f

# Ver logs de un servicio específico
docker compose logs -f backend
docker compose logs -f hapi-fhir    # tarda en arrancar, normal

# Ver logs de los últimos 50 líneas de un servicio
docker compose logs --tail=50 backend

Ejecutar el Seed Script

Crea ≥ 30 pacientes sintéticos desde el dataset real, con Observations LOINC e imágenes en MinIO. Esto vale 0.4 pts.

⚠️
Ejecuta el seed DESPUÉS de que todos los servicios estén healthy, especialmente postgres, hapi-fhir y minio.

📋 Qué debe hacer el seed script

  • Leer filas del dataset PIMA Diabetes (CSV incluido en el repo)
  • Para cada fila, crear un recurso Patient FHIR en HAPI FHIR

💡 Ejemplo de estructura del seed.py

"""
seed.py — Crea pacientes sintéticos desde PIMA Diabetes + APTOS 2019
Ejecutar: python seed/seed.py
"""
import pandas as pd
import requests
import uuid
from minio import Minio
import os, json
from pathlib import Path

# ── Configuración ─────────────────────────────────────────────────
FHIR_BASE   = os.getenv("FHIR_BASE_URL", "http://hapi-fhir:8080/fhir")
DB_URL      = os.getenv("DATABASE_URL")
MINIO_URL   = os.getenv("MINIO_URL", "minio:9000")
MINIO_KEY   = os.getenv("MINIO_ACCESS_KEY")
MINIO_SEC   = os.getenv("MINIO_SECRET_KEY")
BUCKET      = os.getenv("MINIO_BUCKET", "imagenes-clinicas")
DATASET_CSV = Path("seed/pima_diabetes.csv")
IMAGES_DIR  = Path("seed/aptos_sample/")   # 15+ imágenes de muestra

# ── Inicializar MinIO ──────────────────────────────────────────────
minio_client = Minio(MINIO_URL, access_key=MINIO_KEY,
                     secret_key=MINIO_SEC, secure=False)
if not minio_client.bucket_exists(BUCKET):
    minio_client.make_bucket(BUCKET)

# ── Leer dataset ──────────────────────────────────────────────────
df = pd.read_csv(DATASET_CSV).head(40)   # 40 filas → ≥ 30 pacientes

# LOINC codes para PIMA Diabetes
LOINC_MAP = {
    "Glucose":              "2339-0",
    "BloodPressure":        "55284-4",
    "SkinThickness":        "39156-5",
    "Insulin":              "14749-6",
    "BMI":                  "39156-5",
    "DiabetesPedigreeFunction": "90699-3",
    "Age":                  "30525-0",
}

images = list(IMAGES_DIR.glob("*.png"))

for i, row in df.iterrows():
    patient_id = f"pima-{uuid.uuid4().hex[:8]}"

    # 1. Crear Patient en HAPI FHIR
    patient = {
        "resourceType": "Patient",
        "id": patient_id,
        "identifier": [{"system": "urn:pima-diabetes", "value": patient_id}],
        "name": [{"use": "official", "family": f"Paciente{i}", "given": ["Sintético"]}],
        "gender": "female",
        "birthDate": f"{2024 - int(row['Age'])}-01-01",
    }
    r = requests.put(f"{FHIR_BASE}/Patient/{patient_id}", json=patient,
                     headers={"Content-Type": "application/fhir+json"})
    assert r.status_code in [200, 201], f"Error creando Patient: {r.text}"

    # 2. Crear Observations LOINC
    for col, loinc in LOINC_MAP.items():
        obs = {
            "resourceType": "Observation",
            "status": "final",
            "code": {"coding": [{"system": "http://loinc.org",
                                  "code": loinc, "display": col}]},
            "subject": {"reference": f"Patient/{patient_id}"},
            "valueQuantity": {"value": float(row[col]), "unit": "unit"},
        }
        requests.post(f"{FHIR_BASE}/Observation", json=obs,
                      headers={"Content-Type": "application/fhir+json"})

    # 3. Para los primeros 20: subir imagen a MinIO + Media FHIR
    if i < 20 and i < len(images):
        img_path = images[i]
        obj_name = f"fundus/{patient_id}.png"
        minio_client.fput_object(BUCKET, obj_name, str(img_path),
                                 content_type="image/png")
        img_url = f"http://minio:9000/{BUCKET}/{obj_name}"

        media = {
            "resourceType": "Media",
            "status": "completed",
            "subject": {"reference": f"Patient/{patient_id}"},
            "content": {"contentType": "image/png", "url": img_url},
        }
        requests.post(f"{FHIR_BASE}/Media", json=media,
                      headers={"Content-Type": "application/fhir+json"})

    print(f"[{i+1}/40] Paciente {patient_id} creado ✓")

print("\\n✅ Seed completado: 40 pacientes creados")

Configurar Nginx

Nginx es la única puerta de entrada al sistema. Debe tener rate-limiting y headers de seguridad para obtener el puntaje completo.

🏆
La rúbrica evalúa: Nginx como único punto de entrada + rate-limit anti-DDoS + headers de seguridad. Sin esto no se obtienen los 0.7 pts de infraestructura ni los 0.2 pts de seguridad.

📄 nginx/nginx.conf completo

events {
    worker_connections 1024;
}

http {
    # ── Rate Limiting (anti-DDoS) ─────────────────────────────────
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=30r/m;
    limit_req_zone $binary_remote_addr zone=login_limit:10m rate=5r/m;
    limit_conn_zone $binary_remote_addr zone=conn_limit:10m;

    # ── Ocultar versión de Nginx ──────────────────────────────────
    server_tokens off;

    # ── Logs ─────────────────────────────────────────────────────
    access_log /var/log/nginx/access.log;
    error_log  /var/log/nginx/error.log;

    server {
        listen 80;
        server_name _;   # Acepta cualquier dominio / IP

        # ── Headers de Seguridad ──────────────────────────────────
        add_header X-Frame-Options "SAMEORIGIN" always;
        add_header X-Content-Type-Options "nosniff" always;
        add_header X-XSS-Protection "1; mode=block" always;
        add_header Referrer-Policy "strict-origin-when-cross-origin" always;
        add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline';" always;

        # ── Límite de conexiones simultáneas por IP ────────────────
        limit_conn conn_limit 20;

        # ── FRONTEND SPA ──────────────────────────────────────────
        location / {
            limit_req zone=api_limit burst=20 nodelay;
            proxy_pass http://frontend:80;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        }

        # ── BACKEND API ───────────────────────────────────────────
        location /api/ {
            limit_req zone=api_limit burst=10 nodelay;
            proxy_pass http://backend:8000/api/;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_read_timeout 120s;   # para inferencias largas
        }

        # ── ENDPOINT DE LOGIN (más restrictivo) ───────────────────
        location /api/auth/login {
            limit_req zone=login_limit burst=3 nodelay;
            proxy_pass http://backend:8000/api/auth/login;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }

        # ── HAPI FHIR ─────────────────────────────────────────────
        location /fhir/ {
            limit_req zone=api_limit burst=10 nodelay;
            proxy_pass http://hapi-fhir:8080/fhir/;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_read_timeout 30s;
        }

        # ── MINIO (solo lectura pública si decides exponer) ───────
        # location /storage/ {
        #     proxy_pass http://minio:9000/;
        # }

        # ── WebSocket para orquestador (si aplica) ────────────────
        location /ws/ {
            proxy_pass http://orchestrator:8003;
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection "upgrade";
            proxy_set_header Host $host;
            proxy_read_timeout 86400;
        }

        # ── Página de error personalizada ─────────────────────────
        error_page 429 /rate_limit.html;
        location = /rate_limit.html {
            return 429 '{"error":"Demasiadas solicitudes. Intenta en un momento.","code":429}';
            default_type application/json;
        }
    }
}
✅
Con este nginx.conf obtienes: rate-limit (anti-DDoS), headers de seguridad (X-Frame-Options, CSP, etc.), proxy reverso a todos los servicios, y soporte para WebSockets. Todo requerido por la rúbrica.

Errores Comunes y Soluciones

Los problemas más frecuentes al desplegar este stack en un VPS, con solución directa.

Error #1

hapi-fhir se reinicia constantemente

▶

Síntoma: docker compose ps muestra hapi-fhir como restarting

Causa más común: postgres no está ready cuando hapi-fhir intenta conectarse.

# Ver logs de hapi-fhir
docker compose logs hapi-fhir | tail -30

# Solución: esperar a que postgres esté healthy (ya está en el compose con condition: service_healthy)
# Si persiste, revisar que el usuario/password/db de postgres coincidan con el .env
💡
HAPI FHIR tarda 60-90 segundos en inicializar. Es normal que muestre unhealthy al principio. Dale tiempo.
Error #2

El frontend no carga (504 Gateway Timeout)

▶

Causa: el contenedor del frontend no está corriendo o Nginx no puede alcanzarlo.

# Verificar que el frontend está corriendo
docker compose ps frontend

# Ver logs del frontend
docker compose logs frontend

# Verificar que el nombre de red es correcto en nginx.conf
# proxy_pass http://frontend:80  ← el nombre debe ser el del servicio en compose

# Probar conectividad interna
docker compose exec nginx curl http://frontend:80
Error #3

Build del frontend falla (Out of Memory)

▶

Causa: el VPS tiene poca RAM. Node.js necesita ~1.5-2 GB para el build.

# Crear archivo de swap (memoria virtual en disco)
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# Hacer permanente el swap
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

# Verificar
free -h
# Ahora deberías ver Swap: 2G

# Reintentar el build
docker compose build frontend
Error #4

ONNX Runtime falla: "No providers..."

▶

Causa: estás usando onnxruntime-gpu en un servidor sin GPU.

# En requirements.txt del ml-service y dl-service:
# ❌ INCORRECTO
onnxruntime-gpu

# ✅ CORRECTO
onnxruntime

# Reconstruir el servicio tras corregir
docker compose up -d --build ml-service
docker compose up -d --build dl-service
Error #5

Backend no conecta a PostgreSQL

▶

Causa: la DATABASE_URL usa localhost en vez del nombre del servicio.

# ❌ INCORRECTO (localhost no funciona dentro de Docker)
DATABASE_URL=postgresql://user:pass@localhost:5432/db

# ✅ CORRECTO (usar el nombre del servicio definido en compose)
DATABASE_URL=postgresql://user:pass@postgres:5432/db

# También verificar: ¿los valores de POSTGRES_USER/PASSWORD coinciden?
docker compose exec postgres psql -U sduser -d salud_digital_db -c "\l"
Error #6

MinIO: "No such bucket"

▶

Causa: el bucket no fue creado antes de subir imágenes.

# Crear bucket manualmente
docker run --rm \
  --network "$(basename $(pwd))_backend-net" \
  -e MC_HOST_minio="http://minioadmin:PASSWORD@minio:9000" \
  minio/mc mb minio/imagenes-clinicas

# Verificar que existe
docker run --rm \
  --network "$(basename $(pwd))_backend-net" \
  -e MC_HOST_minio="http://minioadmin:PASSWORD@minio:9000" \
  minio/mc ls minio/
⚠️
El nombre de la red Docker por defecto es NOMBRE_CARPETA_backend-net. Si tu carpeta se llama mi-proyecto, la red es mi-proyecto_backend-net.
Error #7

CORS: el frontend no puede llamar al backend

▶

Causa: el backend tiene CORS configurado solo para localhost, no para la IP del VPS.

# En el backend FastAPI (main.py)
from fastapi.middleware.cors import CORSMiddleware
import os

app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "http://localhost:3000",           # desarrollo local
        "http://localhost:5173",           # Vite dev server
        os.getenv("FRONTEND_URL", "*"),   # URL del VPS (desde .env)
    ],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

Y en el .env: FRONTEND_URL=http://IP_DEL_VPS

Error #8

Datos se pierden al hacer docker compose down

▶

Causa: no hay volúmenes nombrados, o se usó down -v por error.

# En docker-compose.yml, los servicios con datos deben tener volúmenes:
services:
  postgres:
    volumes:
      - postgres_data:/var/lib/postgresql/data   # ← volumen nombrado

  minio:
    volumes:
      - minio_data:/data                          # ← volumen nombrado

volumes:
  postgres_data:   # declaración del volumen
  minio_data:
🚨
Nunca uses docker compose down -v en producción. Eso borra todos los volúmenes y pierdes todos los datos.
